ADDITIONAL — EPAM KAFKA Q&A (Resume Mapped — NO Appointment References)
================================================================================

1. Kafka use kyun kiya?
👉 FHIR document sync, consent propagation, aur notifications ke liye async decoupling.
   External system calls slow/unreliable hain — Kafka buffer karta hai, retry + replay possible.

2. Real event flow — Consent Revoked
👉 Patient consent revoke karta hai → consent-service DB update → Kafka "consent-revoked" publish
   → fhir-sync-service: active syncs stop + cache clear
   → ai-assistant-service: patient context remove
   → notification-service: audit log + alert

3. Real event flow — FHIR Document Shared
👉 Provider document share request → consent check (sync) → fhir-sync-service fetch from external
   → DB save + Kafka "document-shared" → notification-service notify recipient
   → AI indexing job: document chunk → embed → pgvector store (async)

4. Topic aur partition kaise decide kiye?
👉 High throughput topics (document-events): 6+ partitions
   Ordering needed per patient: partition key = patientId
   Consumer count ≤ partition count

5. Duplicate messages kaise handle kiye?
👉 Idempotent consumer — processed_event_id table / Redis dedup key
   Same eventId aaye → skip business logic, commit offset

6. Exactly-once vs At-least-once
👉 At-least-once + idempotency — consent revoke dubara process ho to bhi same end state

7. Kafka + DB consistency
👉 Outbox pattern — consent update + event same DB transaction me
   Outbox worker Kafka publish karta hai — guaranteed eventual consistency

8. Consumer lag monitor
👉 CloudWatch MSK metrics (EstimatedMaxTimeLag) → alarm > 5 min → scale consumers

9. Kafka delay impact
👉 Consent revoke delayed = security window (critical — monitor closely)
   FHIR sync delayed = stale documents (acceptable short-term, alert if prolonged)
   Notifications delayed = low impact

10. PHI in Kafka payloads
👉 Payload me sirf reference IDs (patientId, documentId) — no raw PHI
    Consumer full data DB se fetch karta hai after consent re-check

11. Retry + DLQ
👉 3 retries exponential backoff → fail → DLQ topic → manual reprocess after fix

12. Kafka vs REST in your project
👉 REST: consent check, synchronous FHIR fetch response to user
    Kafka: consent propagation, async FHIR sync, document indexing, notifications

13. Project summary (Kafka)
👉 "Kafka use kiya FHIR document sync aur consent event propagation ke liye.
    Consent-revoked events downstream services ko instantly inform karti hain.
    Idempotent consumers + lag monitoring + DLQ se system reliable aur HIPAA-compliant raha."

ADDITIONAL — EPAM AI / FHIR SEARCH Q&A (Resume Mapped)
================================================================================

1. AI assistant ka architecture
👉 User query → consent check → embedding generate (OpenAI) → pgvector semantic search
   on indexed FHIR/clinical documents → top-K context retrieve → prompt pipeline
   → OpenAI chat completion → validation layer → response

2. FHIR documents searchable kaise banaye?
👉 FHIR DocumentReference fetch → extract text content (PDF parse / clinical text)
   → chunk into segments → generate embeddings → store in pgvector table with metadata
   (documentId, patientId, documentType)

3. Semantic search kyun?
👉 User "lab results for diabetes" bole ya "blood sugar test report" — keyword miss karega,
   semantic search same meaning samjhega across FHIR clinical documents

4. pgvector kyun alag vector DB nahi?
👉 Data already PostgreSQL me tha (FHIR metadata + vectors same DB)
   Scale sufficient tha, extra infra cost/complexity avoid

5. Prompt pipeline flow
👉 System prompt (rules: no diagnosis, consent scope only, cite document source)
   + Retrieved FHIR document chunks (dynamic context)
   + User query → LLM → Response validation → Return with source references

6. Hallucination control
👉 Context grounding (sirf retrieved FHIR chunks se answer)
   Temperature=0.3, validation layer, "Information not found in your records" fallback
   Source document reference mandatory in response

7. Sensitive data AI ko kaise?
👉 Consent check before query, PII masked in prompts (reference IDs only)
   Audit log every AI query, no raw SSN/MRN in OpenAI call

8. OpenAI down ho jaye to?
👉 Cached FAQ responses, graceful fallback message, retry with backoff
   Core FHIR document access (non-AI) unaffected — AI is enhancement not blocker

9. AI search vs FHIR REST search ($everything, _search)
👉 FHIR REST = structured exact queries (specific resource types, filters)
   AI search = natural language over document content (semantic, cross-document)
   Dono complementary — FHIR API for precise fetch, AI for intelligent discovery

10. Project summary (AI + FHIR)
👉 "FHIR documents ko index karke pgvector semantic search enable kiya.
    Users natural language me apne health records query kar sakte hain.
    Consent-enforced, context-grounded AI responses with hallucination guardrails."
