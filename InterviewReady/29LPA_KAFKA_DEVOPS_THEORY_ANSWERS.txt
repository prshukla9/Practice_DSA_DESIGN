────────────────────────────────────────────────────────────────────────────────
Q1. Kafka kya hai? Simple language mein explain karo.
────────────────────────────────────────────────────────────────────────────────

Kafka ek distributed messaging system hai — producer messages publish karta hai,
consumer messages consume karta hai. Dono ka direct connection nahi hota.

WhatsApp group analogy:
  - Producer = Jo message bheja (doctor ne patient update publish kiya)
  - Topic = WhatsApp group (patient-events)
  - Partition = Group ke andar alag threads (High priority / Normal priority)
  - Consumer = Jo padh raha hai (notification service, audit service)
  - Broker = WhatsApp server jo messages store karta hai
  - Consumer Group = Ek team jo kaam baant ke karte hain

EPAM project mein:
  Doctor availability update → Kafka topic 'doctor-events' → Notification service consume karta tha
  Patient consent change → 'consent-events' → Audit service + Access control service

Blue Yonder project mein:
  Supply chain data → 'supply-events' → Materialization service consume karta tha
  Order updates → 'order-events' → Downstream analytics service

────────────────────────────────────────────────────────────────────────────────
Q2. Partition kyun use karte hain? Kitni partitions rakhni chahiye?
────────────────────────────────────────────────────────────────────────────────

Partition = Parallelism unit. Jitni partitions utne zyada consumers parallel kaam kar sakte hain.

WHY PARTITION:
  - Ek partition = Ek consumer (same group) → Parallel processing
  - Order guarantee: Messages same partition mein order guaranteed hain
  - Scale: Partitions alag brokers pe spread hoti hain → Distributed storage

KITNI PARTITIONS:
  Formula: max(target throughput / producer throughput, target throughput / consumer throughput)
  Practical rule: Consumers ki expected count ke barabar partitions rakho
  Blue Yonder: 4 consumer instances → 4 partitions for supply-events topic

PARTITION KEY:
  No key → Round-robin (koi ordering guarantee nahi)
  With key → Same key = Same partition (ordering guarantee for that key)

  EPAM: patient_id as key → Ek patient ke saare events same partition mein order mein

────────────────────────────────────────────────────────────────────────────────
Q3. Consumer Group kya hota hai? Rebalancing kab hoti hai?
────────────────────────────────────────────────────────────────────────────────

Consumer Group = Ek group of consumers jo ek topic consume karte hain together.
Rule: Ek partition pe sirf ek consumer (same group) at a time.

Diagram:
  Topic: patient-events (4 partitions)
  Group: notification-group (4 consumers)
  → Consumer A = Partition 0
  → Consumer B = Partition 1
  → Consumer C = Partition 2
  → Consumer D = Partition 3

  Group: audit-group (2 consumers)
  → Consumer X = Partition 0 + 1
  → Consumer Y = Partition 2 + 3

REBALANCING KAB HOTI HAI:
  1. Naya consumer group mein join kiya
  2. Existing consumer crash hua (session timeout)
  3. Topic mein naye partitions add hue
  4. max.poll.interval.ms exceed hua (processing slow tha)

BLUE YONDER P1 INCIDENT (Real story):
  Problem: enable.auto.commit=true tha
  Consumer ne message read kiya → Auto-commit → Crash → Rebalance
  Doosra consumer ne same message process kiya → DUPLICATE!
  100K+ supply records duplicate process ho gaye → Wrong materialization data
  
  Fix: enable.auto.commit=false + manual ACKNOWLEDGE after processing
  Idempotency: processed_event table mein event_id check → already processed? Skip!

COOPERATIVE STICKY REBALANCING:
  Old way: Stop-the-world rebalancing → Sabhi consumers ruk jaate hain
  New way (CooperativeStickyAssignor): Incremental — sirf moved partitions pause
  Blue Yonder mein configure kiya:
    partition.assignment.strategy = CooperativeStickyAssignor
  Result: Rebalancing ke time bhi baaki partitions processing nahi rukti

────────────────────────────────────────────────────────────────────────────────
Q4. At-least-once vs Exactly-once vs At-most-once — kya hai difference?
────────────────────────────────────────────────────────────────────────────────

AT-MOST-ONCE (Fire and forget):
  Producer sends → Ack nahi wait karta → Message lost ho sakta hai
  Use: Logs, metrics jahan loss acceptable hai
  Config: acks=0

AT-LEAST-ONCE (Our default):
  Producer sends → Ack wait karta hai → Failure pe retry
  Risk: Network glitch pe duplicate delivery possible
  Config: acks=all, retries=3
  EPAM + Blue Yonder: Ye use kiya

EXACTLY-ONCE (Hardest):
  Kafka Transactions: enable.idempotence=true + transactional.id + acks=all
  Producer pe idempotency: Broker detect karta hai duplicate → Discard
  But: Sirf Kafka-to-Kafka exactly once guarantee hai
  DB mein write exactly once: Idempotent consumer pattern zaruri hai

PRACTICAL EXACTLY-ONCE (Our approach at Blue Yonder):
  At-least-once Kafka + Idempotent Consumer = Effectively exactly-once
  How: processed_events table mein event_id store karo
       Duplicate aaye → "Event already processed" → Skip
  Yahi production-safe approach hai

────────────────────────────────────────────────────────────────────────────────
Q5. DLQ (Dead Letter Queue) kya hai? Kaise handle kiya?
────────────────────────────────────────────────────────────────────────────────

DLQ = Ek alag topic jahan failed messages jaate hain processing ke baad.

STORY (conveyor belt analogy):
  Factory mein defective items → Alag tray mein jaate hain (DLQ)
  Normal items → Process hote hain
  Defective tray → Engineer baad mein dekhega

FLOW:
  Consumer → Process fail → Retry 3 times (2 sec backoff) → Still fail → DLQ topic mein bhejo
  DLQ topic: 'patient-events.DLT'
  Ops team → DLQ monitor karo → Root cause fix karo → Re-process karo

BLUE YONDER SCENARIO:
  Supply event aaya → Materialization fail (DB timeout) → DLQ
  Alert trigger → On-call engineer notified → DB connection pool fix → Re-process DLQ

SPRING KAFKA CONFIG:
  DeadLetterPublishingRecoverer + FixedBackOff(2000L, 3)
  After 3 retries → topic + ".DLT" mein publish

RE-PROCESS DLQ:
  Admin API: POST /admin/kafka/reprocess-dlq
  Read from DLQ topic → Validate → Republish to original topic
  EPAM: Only Ops role can trigger this (RBAC)

Q6. Consumer Lag kya hai? Blue Yonder mein kaise detect kiya?
────────────────────────────────────────────────────────────────────────────────

Consumer Lag = (Latest Offset produced) - (Current consumer offset)
High lag = Consumers produce hone se peeche hain → Processing slow

BLUE YONDER INCIDENT:
  Kafka consumer lag suddenly 50,000+ messages peak pe gaya
  Root cause: Database connection pool exhausted (10 connections only)
  Each message: 3 DB queries → 10 connections → Queue wait → Slow processing
  
  Detection: kafka-consumer-groups.sh --describe command
             CloudWatch metrics alert (lag > 1000)

  Fix:
    1. DB connection pool 10 → 50 increase kiya (HikariCP maxPoolSize)
    2. Batch processing: 100 messages ek saath process, single batch INSERT
    3. Parallel processing: concurrency=4 (4 concurrent listeners)
  Result: Lag cleared in 15 minutes, processing resumed normal

MONITORING COMMAND:
  kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
    --describe --group supply-chain-group

  Output: TOPIC | PARTITION | CURRENT-OFFSET | LOG-END-OFFSET | LAG

ALERT SETUP:
  CloudWatch custom metric: consumer_lag > 1000 → SNS alert → PagerDuty → On-call

────────────────────────────────────────────────────────────────────────────────
Q7. Outbox Pattern kya hai aur kyun use kiya?
────────────────────────────────────────────────────────────────────────────────

PROBLEM without outbox:
  Step 1: DB mein order save karo (success)
  Step 2: Kafka mein event publish karo (FAIL! Network issue)
  Result: DB mein data hai, Kafka mein event nahi → Inconsistency!

THE 2-PHASE COMMIT PROBLEM:
  DB + Kafka ek distributed transaction mein nahi aate
  XA transactions slow aur complex hain

OUTBOX PATTERN SOLUTION:
  Step 1: DB mein order save karo + outbox table mein event save karo → SAME TRANSACTION
  Step 2: Scheduler (every 100ms) outbox table read karo → Kafka publish karo → Outbox record delete karo
  
  Result: Agar Kafka publish fail → Outbox mein event rahega → Retry karega
          Agar DB fail → Outbox bhi fail → Consistent!

EPAM HEALTHCARE USE CASE:
  Patient consent update → consent table + outbox ek transaction mein save
  Outbox publisher → Kafka 'consent-events' → Audit service + Access service
  Guarantee: Consent change hoga toh event zaroor publish hoga (eventually)

Q8. Kafka vs RabbitMQ — kab kya choose karo?
────────────────────────────────────────────────────────────────────────────────

KAFKA:
  ✅ High throughput (millions/sec)
  ✅ Message retention (replay possible — days/weeks)
  ✅ Consumer groups (multiple consumers, different offsets)
  ✅ Event sourcing, audit log, stream processing
  ✅ Distributed, fault-tolerant
  ❌ Complex setup, Zookeeper/KRaft overhead
  ❌ Message-level acknowledgment complex

USE KAFKA WHEN:
  - Volume bahut high hai (100K+ events/sec)
  - Replay chahiye (audit, analytics, debug)
  - Multiple consumers same data alag-alag process karein
  - Event streaming (real-time dashboard, CDC)

RABBITMQ:
  ✅ Simple setup
  ✅ Complex routing (exchanges, bindings, dead-lettering)
  ✅ Message-level ACK easy
  ✅ Priority queues built-in
  ❌ Messages delete hoti hain after consumption (no replay by default)
  ❌ Lower throughput than Kafka

USE RABBITMQ WHEN:
  - Task queues (background jobs, email sending)
  - Complex routing needed (fanout, topic, direct exchange)
  - RPC (request-reply) pattern
  - Simpler setup preferred

EPAM choice: Kafka — Healthcare data high volume + audit replay requirement
Blue Yonder: Kafka — Supply chain events high volume + materialization streaming

Q9. EPAM mein API latency 200ms kaise reduce ki? (Your resume point)
────────────────────────────────────────────────────────────────────────────────

ORIGINAL PROBLEM:
  Patient dashboard API: Doctor list + Available slots + Patient history
  Sequential DB calls:
    DB Call 1: Fetch doctors (150ms)
    DB Call 2: Fetch available slots (180ms)
    DB Call 3: Fetch patient history (120ms)
    Total: 450ms ← TOO SLOW

DIAGNOSIS (how we found it):
  1. Added @Timed annotation on service methods
  2. Micrometer + Prometheus metrics → Grafana dashboard
  3. Identified: 3 sequential DB calls (no dependency between them!)
  4. N+1 found: For each doctor → separate slot query (JOIN FETCH missing)

FIX 1 — PARALLEL DB CALLS (CompletableFuture):
  Before: Sequential 450ms
  After: Parallel 180ms (longest call time only!)
  
  How: CompletableFuture.supplyAsync() for each DB call
       CompletableFuture.allOf() to wait for all
       Result: All 3 run simultaneously on separate threads

FIX 2 — N+1 ELIMINATION (JOIN FETCH):
  Before: 1 query for doctors + N queries for slots = N+1 problem
  After: 1 query with JOIN FETCH → Single DB round trip

FIX 3 — CACHING (Redis):
  Doctor availability (changes every 30 min) → Redis cache TTL 30 min
  Cache hit: 2ms instead of 180ms DB call
  Cache miss: DB call → Store in Redis → Return

TOTAL IMPROVEMENT:
  Before: 530ms (sequential + N+1)
  After:  ~200ms (parallel + JOIN FETCH + cache)
  Improvement: ~200ms reduction, 30% faster response

HOW TO ANSWER IN INTERVIEW:
  "We profiled the API using Micrometer metrics and found three independent DB calls
  running sequentially. Converted to CompletableFuture parallel calls, fixed N+1 with
  JOIN FETCH, and added Redis caching for doctor availability. Result: 530ms → ~200ms"

Q10. N+1 Problem kya hota hai? Real project mein kaise fix kiya?
────────────────────────────────────────────────────────────────────────────────

N+1 PROBLEM:
  1 query → List of N items fetch
  N queries → Har item ke liye child data fetch
  Total: 1 + N queries (INEFFICIENT!)

EPAM EXAMPLE:
  Query 1: SELECT * FROM doctors WHERE hospital_id = 1 → 50 doctors
  Query 2-51: SELECT * FROM slots WHERE doctor_id = ? (50 times!)
  Total: 51 DB calls for simple page load

FIX 1 — JOIN FETCH (JPA):
  @Query("SELECT d FROM Doctor d JOIN FETCH d.availableSlots WHERE d.hospitalId = :id")
  → Single query with JOIN → All data in one round trip

FIX 2 — @EntityGraph:
  @EntityGraph(attributePaths = {"availableSlots", "specializations"})
  → Spring Data JPA automatically generates JOIN FETCH
  
FIX 3 — @BatchSize (Hibernate):
  @BatchSize(size = 25)
  → Instead of N queries → N/25 batch queries
  → Good when JOIN FETCH not possible (collection too large)

HOW TO DETECT:
  Hibernate SQL logging: spring.jpa.show-sql=true
  p6spy: Shows every SQL with timing
  If you see same query repeated N times → N+1 found!

Q12. Circuit Breaker kya hota hai? Kab use kiya?
────────────────────────────────────────────────────────────────────────────────

PROBLEM WITHOUT CIRCUIT BREAKER:
  Patient service → AI service (slow/down) → 30 sec wait → Thread block → 
  100 requests waiting → Thread pool exhaust → Patient service bhi down!
  Cascade failure — ek service ke down se sab down!

CIRCUIT BREAKER (Electrical analogy):
  CLOSED state: Normal — requests flow through
  OPEN state: AI service fail ho rahi hai → Circuit trip → Immediately fail fast (no wait)
  HALF-OPEN state: Kuch time baad → Test karo ek request se → Service back? → Close karo

THREE STATES:
  CLOSED → 5 failures in 10 sec → OPEN
  OPEN → 30 sec baad → HALF-OPEN
  HALF-OPEN → Success → CLOSED
  HALF-OPEN → Failure → OPEN again

EPAM PROJECT:
  Patient service → OpenAI API call
  OpenAI slow/down → Circuit breaker OPEN → Fallback: "AI service unavailable, try later"
  Patient service nahi ruka — fast failure, thread free

RESILIENCE4J CONFIG (Key numbers yaad rakho):
  failureRateThreshold = 50% → 50% requests fail → Open
  waitDurationInOpenState = 30 seconds
  permittedNumberOfCallsInHalfOpenState = 5

FALLBACK STRATEGY:
  @CircuitBreaker(name = "ai-service", fallbackMethod = "fallbackResponse")
  Fallback: Return cached response / default message / degrade gracefully

Q13. Saga Pattern — Distributed Transaction without 2-Phase Commit
────────────────────────────────────────────────────────────────────────────────

PROBLEM:
  Patient appointment booking: 
    Step 1: Doctor availability check + reserve
    Step 2: Payment charge
    Step 3: Appointment confirmation email
  
  Agar Step 2 fail ho → Step 1 ka reservation rollback kaise karein?
  Traditional 2-Phase Commit (2PC): Lock sab services → Slow → Deadlock risk

SAGA CHOREOGRAPHY (Our approach — EPAM):
  Each service listens to events + publishes events
  No central coordinator

  HAPPY PATH:
    1. Booking Service → Publish 'appointment.initiated'
    2. Doctor Service → Listen → Reserve slot → Publish 'slot.reserved'
    3. Payment Service → Listen → Charge → Publish 'payment.success'
    4. Notification Service → Listen → Send email → Publish 'appointment.confirmed'

  FAILURE PATH (Compensating transactions):
    3. Payment Service → Charge FAIL → Publish 'payment.failed'
    4. Doctor Service → Listen 'payment.failed' → RELEASE slot (compensating!)
    5. Booking Service → Mark CANCELLED

COMPENSATION = Undo operation:
  Slot reserve → compensation = Slot release
  Payment charge → compensation = Payment refund

WHEN SAGA IS HARD:
  Cross-cutting transactions — multiple compensations needed
  Long-running sagas — state tracking complex
  Solution: Saga orchestrator (central service manages state)

Q15. HIPAA Compliance — Kya implement kiya?
────────────────────────────────────────────────────────────────────────────────

HIPAA = Health Insurance Portability and Accountability Act
PHI = Protected Health Information (patient name, DOB, SSN, diagnosis)

EPAM IMPLEMENTATION (3 areas):

1. AUDIT LOGGING:
   Every PHI access → Who accessed, when, what data, from where
   @HipaaAudit custom annotation → AOP @Around aspect

   Purpose: Agar PHI leak ho → Trace back karo kisne access kiya

2. FIELD ENCRYPTION:
   PHI fields (SSN, DOB, diagnosis) → AES-256 encrypt in DB
   JPA AttributeConverter: DB mein encrypted → Java object mein decrypted
   Key management: AWS KMS (Key Management Service)

3. CONSENT MANAGEMENT:
   Patient ne consent diya hai? → Agar nahi → Access denied
   Consent scope: "Dr. Smith can see my cardiac records only"
   Consent check → AOP interceptor → Before every PHI access

INTERVIEW ANGLE:
  "We took a layered approach to HIPAA compliance:
   AES-256 field encryption for PHI at rest,
   AOP-based audit logging for every access,
   and consent management to ensure data is only accessed by authorized parties.
   This gave us defense-in-depth for patient data protection."

Q16. Materialization kya hai? Blue Yonder mein kaise kiya?
────────────────────────────────────────────────────────────────────────────────

MATERIALIZATION = Pre-computed aggregates store karna taaki query fast ho.

PROBLEM (Blue Yonder):
  Supply chain dashboard: "Show total orders per region per day"
  Naive: SELECT sum(orders) GROUP BY region, date → Crores rows scan → SLOW!

MATERIALIZED VIEW APPROACH:
  Pre-compute aggregates → Materialized table mein store
  Dashboard query: SELECT * FROM materialized_supply_summary → Fast!

INCREMENTAL UPDATE (Kafka-driven):
  New order event → Kafka → Materialization service consume
  Instead of full recompute → Just update affected region's count
  e.g. Order for Region NORTH → materialized_supply_summary WHERE region='NORTH' UPDATE count+1

NIGHTLY RECONCILIATION:
  Incremental updates sometimes miss due to errors
  Nightly batch: Full recompute from raw data → Compare with materialized → Fix discrepancies
  @Scheduled(cron = "0 2 * * *") → 2 AM every night

BENEFITS:
  Dashboard query: 45 seconds → 200ms (225x faster!)
  DB load: Raw table 10 crore rows vs Materialized table 1000 rows

INTERVIEW ANSWER:
  "In Blue Yonder, we implemented materialization for supply chain analytics.
  Kafka events triggered incremental updates to pre-computed aggregate tables,
  dramatically reducing query time from 45 seconds to under 200ms.
  A nightly reconciliation job ensured data consistency with the source of truth."

Q17. Blue-Green Deployment kya hota hai? Canary se kaise alag?
────────────────────────────────────────────────────────────────────────────────

BLUE-GREEN DEPLOYMENT:
  Two identical environments: Blue (current live) + Green (new version)
  
  Process:
    1. Green environment mein naya code deploy karo
    2. Green pe tests run karo (staging)
    3. Load balancer switch: Blue → Green (instant cutover!)
    4. Problem aaye → Rollback: Green → Blue (instant!)
    5. Blue environment → Next release ke liye ready

  Benefits: Zero downtime, instant rollback
  Cost: 2x infrastructure (two environments maintain karna padta hai)

CANARY DEPLOYMENT:
  Naya version sirf kuch % traffic pe release karo
  
  Process:
    1. v2 deploy karo (1% traffic → v2, 99% → v1)
    2. Monitor metrics (errors, latency, business KPIs)
    3. Sab theek? → 10% → 25% → 50% → 100% gradually
    4. Problem aaye → Route 100% back to v1

  Benefits: Real traffic pe test, gradual risk exposure
  Cost: Versioning complexity, monitoring zaroori

WHEN TO USE WHICH:
  Blue-Green: Full release with rollback safety (critical features)
  Canary: Gradual rollout ke saath monitoring (high-risk changes, ML models)

TARGET JD MEIN YE MENTION THA → Ye zaroor poochha jaayega!

Q18. Docker vs Kubernetes — kab kya aur kyun?
────────────────────────────────────────────────────────────────────────────────

DOCKER:
  Container create karta hai — Application + dependencies bundled
  "Works on my machine" problem solve
  Single container manage karta hai

KUBERNETES (K8s):
  Multiple containers orchestrate karta hai
  Auto-healing, auto-scaling, load balancing — sab built-in
  "Production pe kaise manage karein?" — K8s solve karta hai

ANALOGY:
  Docker = Shipping container (ek box mein sab kuch pack)
  K8s = Port authority (hazaaron containers manage, track, route)

K8s KEY CONCEPTS:
  Pod = Ek ya zyada containers ka group (smallest deployable unit)
  Deployment = Pod ka desired state define karo (3 replicas chahiye)
  Service = Pods ko network expose karo (stable IP)
  ConfigMap = Non-sensitive config store (DB URL, Kafka broker)
  Secret = Sensitive config store (passwords, API keys) — base64 encoded
  HPA = Horizontal Pod Autoscaler — CPU > 70% → Scale out automatically
  Ingress = External traffic ko services mein route karo (L7 load balancing)

EPAM PRODUCTION SETUP:
  EKS (AWS managed Kubernetes)
  Patient service: 3 replicas, HPA (CPU 70% → max 10 pods)
  Liveness probe: /actuator/health → Pod unhealthy? K8s restart karo
  Readiness probe: /actuator/health/readiness → Not ready? Traffic mat bhejo
  Rolling update strategy: 1 pod at a time update → Zero downtime

Q20. Observability — Monitoring kaise kiya? (ELK + Prometheus + Tracing)
────────────────────────────────────────────────────────────────────────────────

3 PILLARS OF OBSERVABILITY:
  1. METRICS — Kya ho raha hai? (numbers)
  2. LOGS — Kyun ho raha hai? (text)
  3. TRACES — Kahan ho raha hai? (request path)

1. METRICS (Prometheus + Grafana):
  Micrometer → JVM metrics expose karta hai (/actuator/metrics)
  Prometheus → Metrics scrape karta hai every 15 sec
  Grafana → Dashboard visualize karta hai

  Key metrics we tracked:
    - API response time (P50, P95, P99)
    - Error rate (HTTP 5xx / total requests)
    - JVM heap usage (alert if > 80%)
    - Kafka consumer lag (alert if > 1000)
    - DB connection pool usage

  ALERT RULE example:
    IF http_error_rate > 5% for 5 minutes → PagerDuty alert → On-call engineer

2. LOGS (ELK Stack):
  E = Elasticsearch (log storage + search)
  L = Logstash (log aggregation + parsing)
  K = Kibana (log visualization + query)
  
  Logback → JSON format logs → Logstash → Elasticsearch → Kibana
  
  Structured logging (JSON):
    { "timestamp": "...", "level": "ERROR", "traceId": "abc123",
      "service": "patient-service", "userId": "U123", "message": "DB timeout" }
  
  WHY JSON: Kibana mein filter karo: service="patient-service" AND level="ERROR"
  WHY traceId: Distributed tracing se connect karo

3. DISTRIBUTED TRACING (Sleuth + Zipkin/Jaeger):
  Problem: Patient service → AI service → DB — slow ho raha hai, kahan?
  
  Spring Sleuth: Auto-generate traceId for each request
  traceId propagate hota hai across service calls (HTTP headers + Kafka headers)
  
  Zipkin dashboard: Ek request ka full journey dikhao
    Patient API (50ms) → AI Service (800ms ← BOTTLENECK!) → Response
  
  EPAM: Zipkin use kiya → AI API calls slow → OpenAI throttling detect kiya → Caching add kiya

INTERVIEW ANSWER:
  "We had three layers of observability. Prometheus + Grafana for metrics with alerts
  on P95 latency and error rates. ELK stack for structured JSON logs with traceId
  correlation. Spring Sleuth + Zipkin for distributed tracing to identify bottlenecks
  across service calls. This helped us reduce mean time to detect (MTTD) from hours to minutes."

────────────────────────────────────────────────────────────────────────────────
Q21. RAG Pipeline kya hai? Kaise implement kiya?
────────────────────────────────────────────────────────────────────────────────

RAG = Retrieval Augmented Generation
Problem: GPT-4 ko hospital-specific data nahi pata → Hallucinate karta hai
Solution: Pehle relevant data retrieve karo → Context deo → Then GPT-4 answer kare

OUR RAG PIPELINE (EPAM):

  Step 1 — USER QUERY:
    "What medications did Patient X take last month?"

  Step 2 — EMBED QUERY:
    OpenAI text-embedding-ada-002 → Query ko 1536-dimension vector mein convert
    
  Step 3 — SEMANTIC SEARCH (pgvector):
    PostgreSQL + pgvector extension
    Cosine similarity search: Vector DB se top 5 relevant records find karo
    SQL: SELECT * FROM patient_records ORDER BY embedding <=> query_vector LIMIT 5

  Step 4 — BUILD CONTEXT:
    Top 5 records ko text mein format karo
    Context window = System prompt + Patient records + User question

  Step 5 — GPT-4 CALL:
    System: "You are a medical assistant. Only answer based on provided context."
    User: Context + Question → GPT-4 → Answer

  Step 6 — VALIDATE + RETURN:
    Confidence threshold check (similarity score < 0.7 → "Not enough data to answer")
    HIPAA: Response mein sensitive data check karo before sending

PGVECTOR SPECIFICS:
  HNSW index: Fast approximate nearest neighbor search
  IVFFlat index: Better for large datasets (train on existing data)
  Distance operator: <=> (cosine), <-> (L2), <#> (inner product)

────────────────────────────────────────────────────────────────────────────────
Q22. Hallucination kaise rokते hain AI responses mein?
────────────────────────────────────────────────────────────────────────────────

HALLUCINATION = AI confidently wrong answer deta hai

OUR PREVENTION STRATEGIES:

1. SIMILARITY THRESHOLD:
   pgvector search → similarity score < 0.7 → Context unreliable
   Response: "I don't have enough information to answer accurately"
   Better to say "I don't know" than give wrong medical info!

2. PROMPT ENGINEERING:
   System prompt: "ONLY answer using the provided context. If answer not in context, say 'I don't know'"
   "Do NOT make assumptions or use general medical knowledge"

3. RESPONSE VALIDATION:
   Key medical terms extract karo from response
   Check: Are these terms present in the context we provided?
   Suspicious if AI mentions drugs not in patient's records

4. TEMPERATURE = 0:
   GPT temperature: 0 = Deterministic (factual), 1 = Creative
   Healthcare: Always temperature=0 → Reproducible, factual answers

5. HUMAN REVIEW FOR HIGH-RISK:
   Diagnosis suggestions → "Please consult your doctor" disclaimer
   Medication advice → Flag for doctor review in workflow

Q23. P1 Production Issue — Bata ek story jab system down hua
────────────────────────────────────────────────────────────────────────────────

BLUE YONDER P1 — KAFKA DUPLICATE PROCESSING (Ye zaroor poochho bologe):

INCIDENT:
  Date: Production Monday morning (supply chain peak time)
  Alert: "Duplicate supply records detected in materialization table"
  Impact: 100K+ orders showing wrong inventory counts → Business halt

INVESTIGATION (RCA — Root Cause Analysis):
  Step 1: CloudWatch logs check kiye → Kafka consumer rebalancing bar bar ho rahi thi
  Step 2: Consumer config dekhte → enable.auto.commit=true!
  Step 3: Timeline reconstruct:
    Consumer → Read message (offset auto-committed immediately)
    Consumer → Processing karne laga (DB insert)
    Processing ke beech → Application crash → Rebalance
    New consumer → Same message phir read kiya (offset already committed tha kafka mein, but processing complete nahi hua!)
    Wait — auto-commit commit karta hai on poll(), not on processing completion!
    Result: Message processed twice → Duplicate records

FIX (deployed in 2 hours with rollback ready):
  enable.auto.commit=false
  AckMode.MANUAL_IMMEDIATE → After successful processing → Manual acknowledge
  processed_events table → event_id unique constraint → Idempotency

PREVENTIVE MEASURES:
  Post-incident: Unit test likha for idempotency
  Monitoring: Duplicate detection alert in Grafana
  Documentation: Kafka consumer configuration checklist added

HOW TO NARRATE:
  STAR format: Situation (what happened) → Task (what was my role) →
  Action (what I did) → Result (what was the outcome)

Q25. Service Discovery kya hota hai?
────────────────────────────────────────────────────────────────────────────────

PROBLEM:
  Service A → Service B ko call karna hai
  But Service B ka IP address dynamic hai (K8s mein pods restart pe IP change)
  Hard-coded IP → Broken!

SOLUTION — Service Discovery:
  SERVICE REGISTRY: Central place jahan services apna IP register karte hain
  
  Client-side discovery: Service A khud registry query karta hai → Load balance karta hai
  Server-side discovery: Load balancer registry query karta hai → Route karta hai

TOOLS:
  Eureka (Netflix OSS): Spring Cloud Netflix → Self-registration, heartbeat
  K8s built-in: Service resource → DNS name → kube-proxy → Pod routing
  AWS Cloud Map: Managed service discovery

EPAM APPROACH:
  K8s use kiya → K8s Service resource = Built-in service discovery
  patient-service:8080 → K8s DNS → Healthy pod pe route
  No separate Eureka needed when using K8s!

────────────────────────────────────────────────────────────────────────────────
Q26. API Gateway kya kaam karta hai? Spring Cloud Gateway vs NGINX?
────────────────────────────────────────────────────────────────────────────────

API GATEWAY = Single entry point for all client requests.

WHAT IT DOES:
  1. Routing: /patient → Patient Service, /doctor → Doctor Service
  2. Authentication: JWT token validate karo (centralized, not per service)
  3. Rate limiting: Per user/IP limits
  4. SSL termination: HTTPS → HTTP (internal services pe SSL overhead nahi)
  5. Request/Response transformation: Headers add/remove
  6. Load balancing: Multiple instances ke beech distribute

SPRING CLOUD GATEWAY vs NGINX:
  Spring Cloud Gateway:
    Java-based, reactive (WebFlux)
    Custom filters easy in Java
    Spring ecosystem integration (Eureka, Config Server)
    Good for complex business logic in routing

  NGINX:
    High performance, battle-tested
    Static config (or Lua scripting for dynamic)
    Very low memory footprint
    Good for simple reverse proxy + SSL termination

EPAM: Kong API Gateway (based on NGINX + Lua) use kiya
      Why: Built-in plugins (rate limiting, JWT, logging)
