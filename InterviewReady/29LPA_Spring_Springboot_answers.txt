Q1. IOC (Inversion of Control) kya hota hai?
------------------------------------------------------------------------

CONCEPT:
Normal programming mein TUM object create karte ho:
  UserService service = new UserService(new UserRepository());

IOC mein Spring khud objects create karta hai aur tumhe de deta hai.
Control tumhare haath se Spring ke haath mein aa jaata hai — isliye "Inversion"

ANALOGY:
- Bina IOC: Tum khud restaurant jaate ho, order karte ho, khana laate ho
- IOC ke saath: Zomato (Spring) sab handle karta hai — tumhe sirf batana hai kya chahiye

// WITHOUT IOC — tightly coupled, hard to test
public class OrderService {
    // OrderService khud UserRepository create kar raha hai — BAD!
    private UserRepository repo = new UserRepository(); // direct dependency

    public void placeOrder(Order order) {
        User user = repo.findById(order.getUserId());
    }
}

// WITH IOC — Spring manages the dependency
@Service
public class OrderService {
    private final UserRepository repo; // Interface type — loosely coupled

    // Spring isko inject karega — tum mat banao
    @Autowired
    public OrderService(UserRepository repo) {
        this.repo = repo;
    }
}

INTERVIEW TIP: "IOC ek principle hai, Dependency Injection uska implementation hai. Spring IoC container (ApplicationContext) beans create, configure aur manage karta hai."

Q2. Dependency Injection ke types: Constructor vs Setter vs Field
------------------------------------------------------------------------

// TYPE 1: CONSTRUCTOR INJECTION (RECOMMENDED — BEST PRACTICE)
@Service
public class PaymentService {
    private final PaymentGateway gateway; // final — immutable after construction
    private final AuditService audit;

    // Spring constructor call karega aur dependencies inject karega
    // @Autowired optional hai Spring 4.3+ mein agar single constructor ho
    @Autowired
    public PaymentService(PaymentGateway gateway, AuditService audit) {
        this.gateway = gateway;
        this.audit = audit;
    }
}

// WHY CONSTRUCTOR INJECTION BEST HAI:
// 1. final fields — immutable, thread-safe
// 2. Circular dependency early catch hoti hai (runtime nahi — startup pe)
// 3. Testing easy — new PaymentService(mockGateway, mockAudit) seedha karo
// 4. Required dependencies clearly visible hain

// TYPE 2: SETTER INJECTION (optional dependencies ke liye)
@Service
public class NotificationService {
    private EmailClient emailClient; // optional — ho sakta hai configure na ho

    @Autowired(required = false) // Optional dependency
    public void setEmailClient(EmailClient emailClient) {
        this.emailClient = emailClient;

// TYPE 3: FIELD INJECTION (@Autowired directly on field) — AVOID IN PRODUCTION
@Service
public class UserService {
    @Autowired // ❌ Bad practice — final nahi kar sakte, testing mushkil
    private UserRepository userRepository;

    // Testing mein: reflection chahiye ya Spring context chahiye
    // "I avoid field injection because it hides dependencies and makes testing harder"
}

COMPARISON:
| Feature            | Constructor      | Setter          | Field          |
|--------------------|------------------|-----------------|----------------|
| Immutability       | Yes (final)      | No              | No             |
| Circular dependency| Fails fast       | Allowed (risky) | Allowed (risky)|
| Testing            | Easy (new())     | Medium          | Hard           |
| Recommended        | YES              | Optional deps   | NO             |

// PRACTICAL EXAMPLE — hooks ka use
@Component
public class DatabaseConnectionPool {
    private Connection[] connections;

    // Called after all dependencies injected — initialization yahan karo
    @PostConstruct
    public void initialize() {
        System.out.println("Initializing connection pool...");
        connections = createConnections(10); // DB connections setup
    }

    // Called before bean destroyed — cleanup yahan karo
    @PreDestroy
    public void cleanup() {
        System.out.println("Closing connection pool...");
        for (Connection conn : connections) {
            conn.close(); // Connections properly close karo
        }
    }
}

// @Bean annotation ke saath
@Configuration
public class AppConfig {
    @Bean(initMethod = "start", destroyMethod = "stop")
    public CacheManager cacheManager() {
        return new RedisCacheManager();
    }
}

BeanFactory is the basic IoC container responsible only for creating and managing beans,
 and it usually creates beans lazily. ApplicationContext extends BeanFactory and provides enterprise features
  such as automatic bean wiring, component scanning, AOP support, event publishing, internationalization,
   environment and profile support, and web integration.
 That's why almost all modern Spring Boot applications internally use ApplicationContext rather than BeanFactory.

| Feature                | BeanFactory              | ApplicationContext          |
|------------------------|--------------------------|-----------------------------|
| Bean creation          | Lazy (jab maango tab)    | Eager (startup pe hi sab)   |
| Annotations support    | Basic                    | Full (@Autowired etc.)      |
| AOP support            | No                       | Yes                         |
| i18n (messages)        | No                       | Yes (MessageSource)         |
| Events                 | No                       | Yes (ApplicationEvents)     |
| Web support            | No                       | Yes (WebApplicationContext) |
| Use in production      | Rarely                   | ALWAYS                      |

// ApplicationContext implementations:
// AnnotationConfigApplicationContext — Java config
// ClassPathXmlApplicationContext — XML config
// WebApplicationContext — Web apps

// PROGRAMMATIC ACCESS (rare but asked in interviews)
@Component
public class SomeBean implements ApplicationContextAware {
    private ApplicationContext context;

    @Override
    public void setApplicationContext(ApplicationContext context) {
        this.context = context;
    }

    public void doWork() {
        // Dynamic bean retrieval
        UserService service = context.getBean(UserService.class);

}

Q6. Bean Scopes kya hote hain?

// SCOPE 1: SINGLETON (DEFAULT) — Ek bean instance for entire application
@Component
@Scope("singleton") // Default — ye likhna zaroori nahi
public class ConfigService {
    // Ek hi instance — sabhi requests same object use karte hain
    // STATE MAT RAKHO! Thread-safety issue hogi
}

// SCOPE 2: PROTOTYPE — Har baar naya instance
@Component
@Scope("prototype") // Ya @Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
public class OrderProcessor {
    private List<Item> items = new ArrayList<>(); // State rakho — har request ka alag object
}

// SCOPE 3: REQUEST — Ek HTTP request = ek instance (Web only)
@Component
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext {
    private String requestId = UUID.randomUUID().toString();
}

// SCOPE 4: SESSION — Ek HTTP session = ek instance (Web only)
@Component
@Scope(value = "session", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class UserSession {
    private String userId;
    private List<String> cart = new ArrayList<>();
}

// SCOPE 5: APPLICATION — ServletContext = ek instance (rarely used)
// SCOPE 6: WEBSOCKET — WebSocket session scope

INTERVIEW TIP: "proxyMode = TARGET_CLASS SCOPED_PROXY kyun chahiye?
 Kyunki singleton bean mein prototype/request/session bean inject karo toh scope mismatch hota hai.
  Proxy yeh solve karta hai — singleton proxy ko call karo, proxy naya instance return karta hai."

Q7. Singleton Bean Thread-safe hota hai ya nahi?
------------------------------------------------------------------------

SHORT ANSWER: NO! Singleton bean thread-safe NAHI hota by default.

// PROBLEM: Singleton bean with MUTABLE STATE
@Service
public class CounterService {
    private int count = 0; // SHARED STATE — DANGER!

    // Multiple threads simultaneously yahan aayenge
    public int increment() {
        return ++count; // RACE CONDITION! Not atomic!
    }
}

// SOLUTION 1: Stateless banao (BEST)
@Service
public class OrderService {
    // Koi instance variables nahi — stateless — automatically thread-safe!
    public Order processOrder(Order order) {
        // Local variables use karo — each thread ka apna stack
        Order processed = applyDiscount(order);
        return processed;
    }
}

// SOLUTION 2: AtomicInteger use karo
@Service
public class SafeCounterService {
    private AtomicInteger count = new AtomicInteger(0); // Thread-safe

    public int increment() {
        return count.incrementAndGet(); // Atomic operation
    }
}

// SOLUTION 3: Synchronized (performance cost)
@Service
public class SyncService {
    private int count = 0;

    public synchronized int increment() {
        return ++count;
    }
}

Interview me agar pooche:

What is a stateless bean?

Tum confidently bolna:

A stateless bean does not store request-specific or mutable data in instance variables. 
It only works on method parameters and local variables. Since Spring singleton beans are shared across multiple threads, 
keeping them stateless makes them naturally thread-safe because
 local variables are stored in each thread's own stack, while instance variables are shared through the singleton object in the heap.
// SOLUTION 4: ThreadLocal (per-thread state)
@Service
public class RequestScopedService {
    private ThreadLocal<String> currentUser = new ThreadLocal<>();

    public void setUser(String user) { currentUser.set(user); }
    public String getUser() { return currentUser.get(); }
    public void clear() { currentUser.remove(); } // Memory leak avoid!
}

INTERVIEW TIP: "6 yrs exp se expect hai: Singleton beans stateless hone chahiye. Agar state chahiye — prototype scope, request scope, ya ThreadLocal use karo."

Q8. Prototype Bean ka lifecycle kaise manage hota hai?
------------------------------------------------------------------------

IMPORTANT GOTCHA: Spring prototype bean ka @PreDestroy CALL NAHI KARTA!
Spring sirf creation manage karta hai, destruction nahi.

@Component
@Scope("prototype")
public class ReportGenerator {
    private File tempFile;

    @PostConstruct
    public void init() {
        tempFile = createTempFile(); // Creation — Spring karta hai
    }

    @PreDestroy
    public void cleanup() {
        // ⚠️ WARNING: Yeh method prototype scope mein CALL NAHI HOGA!
        // Tumhe manually cleanup karna hoga
        tempFile.delete();
    }
}

// MANUAL CLEANUP: BeanPostProcessor ya DisposableBean implement karo
// Ya simply manually destroy karo:
@Service
public class ReportService {
    @Autowired
    private ApplicationContext context;

    public void generateReport() {
        ReportGenerator generator =
            context.getBean(ReportGenerator.class); // Naya instance har baar
        try {
            generator.generate();
        } finally {
            // Manual cleanup
            ((DisposableBean) generator).destroy();
        }
    }
}

Q9. Lazy Initialization kya hota hai?
------------------------------------------------------------------------

// DEFAULT: Spring sari beans startup pe create karta hai (eager)
// LAZY: Bean tabhi create ho jab pehli baar use ho

// METHOD 1: @Lazy on bean
@Component
@Lazy // Pehli baar inject hone ya getBean() call hone pe create hoga
public class HeavyReportingService {
    public HeavyReportingService() {
        // Expensive initialization — DB load, ML model load etc.
        System.out.println("HeavyReportingService created!");
    }
}

// METHOD 2: @Lazy on injection point
@Service
public class DashboardService {
    @Autowired
    @Lazy // Sirf yahan inject hone pe create hoga
    private HeavyReportingService reportingService;
}

// METHOD 3: Global lazy init (Spring Boot 2.2+)
// application.properties:
// spring.main.lazy-initialization=true
// Saari beans lazy ho jaayengi — startup time fast but first request slow

WHEN TO USE:
- Optional features jo rarely use hoti hain
- Heavy beans (ML models, large caches)
- Faster application startup chahiye

WHEN TO AVOID:
- Critical beans jo startup pe hi validate hone chahiye
- Circular dependencies — lazy se temporarily solve hota hai but wrong approach

// Already covered in Q3 — additional points:

// @PostConstruct ORDER when inheritance:
// Parent @PostConstruct pehle, then Child @PostConstruct

// MULTIPLE @PostConstruct — ONLY ONE allowed per class (JSR-250)

// BeanFactoryPostProcessor — bean definitions modify karo (bean create hone se pehle)
@Component
public class CustomBeanFactoryPostProcessor implements BeanFactoryPostProcessor {
    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory factory) {
        // Bean definitions modify karo — rare but powerful
        BeanDefinition def = factory.getBeanDefinition("userService");
        def.setScope("prototype"); // Programmatically scope change karo
    }
}

// BeanPostProcessor — har bean ke creation pe hook
@Component
public class LoggingBeanPostProcessor implements BeanPostProcessor {
    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        System.out.println("Bean created: " + beanName);
        return bean; // Return bean as-is (ya wrapped proxy)
    }
}

Q11. Spring Boot kya problem solve karta hai?
------------------------------------------------------------------------

PROBLEMS BEFORE SPRING BOOT:
1. Boilerplate XML configuration bahut zyada
2. Dependency version conflicts (Maven hell)
3. Server deployment alag — WAR file bana ke Tomcat mein deploy karo
4. Testing setup complex

SPRING BOOT SOLUTIONS:
1. Auto-configuration — sensible defaults, zero XML
2. Starter dependencies — spring-boot-starter-web ek mein sab aa jaata hai
3. Embedded server — JAR run karo, Tomcat andar hai
4. Production-ready — Actuator, Health checks, Metrics built-in

@ConditionalOnMissingBean(DataSource.class) // Sirf tabhi agar tumne DataSource nahi banaya

PART 1 - ApplicationContext (Spring ka Brain)

Definition

ApplicationContext is the Spring IoC Container.

Simple words: ApplicationContext ek manager hai jo application ke saare
beans ko create, manage, connect aur destroy karta hai.

Movie Analogy

Actors: - UserService - OrderService - PaymentService - Repository -
Controller

Manager: ApplicationContext

Actors ek dusre ko directly nahi dhoondte. Manager bolta hai: - Ye lo
UserRepository - Ye lo PaymentService - Ye lo DataSource

Ye manager hi ApplicationContext hai.

Startup Flow

main() | v SpringApplication.run() | v ApplicationContext Create | +–>
Component Scan +–> Auto Configuration +–> Bean Creation +–> Dependency
Injection +–> Bean Lifecycle +–> AOP Proxy Creation | v Application
Ready

ApplicationContext Responsibilities

1.  Create Beans
2.  Inject Dependencies
3.  Manage Bean Lifecycle
4.  Create AOP Proxies
5.  Register Auto-configured Beans
6.  Return Beans using context.getBean()

BeanFactory vs ApplicationContext

BeanFactory: - Only creates and returns beans. - Lightweight. - Mostly
lazy initialization.

ApplicationContext: - BeanFactory + - Dependency Injection - AOP -
Events - Profiles - Environment - Internationalization - Auto
Configuration

Interview Answer

ApplicationContext is Spring’s central IoC container. During
SpringApplication.run(), Spring creates the ApplicationContext, scans
components, performs auto-configuration, creates beans, injects
dependencies, manages the bean lifecycle, creates AOP proxies when
required and stores all managed beans.

PART 2 - @SpringBootApplication

@SpringBootApplication is a shortcut for:

@SpringBootConfiguration @EnableAutoConfiguration @ComponentScan

Meaning

@SpringBootConfiguration - Marks the main configuration class. -
Basically @Configuration.

@ComponentScan - Scans current package and sub-packages. - Finds:
@Component @Service @Repository @Controller

@EnableAutoConfiguration - Looks at project dependencies and
properties. - Automatically creates missing infrastructure beans.

Example

Dependency: mysql-connector-j

Property: spring.datasource.url=…

Spring automatically creates: - DataSource - JdbcTemplate -
TransactionManager

Internal Flow

SpringApplication.run() | v Create ApplicationContext | v Read
@SpringBootApplication | +–> @SpringBootConfiguration | +–>
@ComponentScan | +–> @EnableAutoConfiguration | v
AutoConfigurationImportSelector | v Read AutoConfiguration.imports | v
Check @Conditional annotations | v Create required beans | v Application
Ready

Interview Answer

@SpringBootApplication combines @SpringBootConfiguration, @ComponentScan
and @EnableAutoConfiguration. During startup Spring scans components,
loads configuration and automatically creates infrastructure beans such
as DataSource, DispatcherServlet and ObjectMapper based on classpath
dependencies and configuration properties.

PART 3 - AOP (Aspect Oriented Programming)

Problem

Every service contains:

Logging Security Transaction

These are repeated in every method.

Business Logic

-   Transfer Money
-   Create User
-   Delete Order

Cross Cutting Concerns

-   Logging
-   Security
-   Transactions
-   Caching
-   Auditing
-   Performance Monitoring

Spring’s Solution

Developer writes only business logic. Spring handles cross-cutting
concerns automatically.

How?

Using PROXY.

Flow

Client | v Proxy | +–> Logging +–> Security +–> Transaction Begin | v
Real Method | +–> Commit | v Return

If exception: Rollback

Example

@Transactional public void transferMoney() { // Business logic only }

Spring Proxy internally performs:

Begin Transaction | Real Method | Commit / Rollback

Main Terms

Aspect - Logging - Security - Transaction

Join Point - Method execution

Advice - Before - After - Around - AfterThrowing

Pointcut - Which methods should receive advice

Proxy - Object created by Spring which intercepts method calls.

Interview Answer

Aspect Oriented Programming separates cross-cutting concerns such as
logging, transactions, security and caching from business logic. Spring
creates a proxy around the target bean. Whenever a method is invoked,
the proxy executes additional behavior before and/or after delegating
the call to the real business method.

Q17. @Autowired vs @Inject vs @Resource?
------------------------------------------------------------------------

| Feature          | @Autowired (Spring) | @Inject (JSR-330) | @Resource (JSR-250) |
|------------------|---------------------|-------------------|---------------------|
| Source           | Spring              | Java Standard     | Java Standard       |
| By type default  | Yes                 | Yes               | No (by name)        |
| required attr    | @Autowired(required=false) | No       | No                  |
| @Qualifier needed| Yes (ambiguity pe)  | Yes               | name attribute      |
| Recommended      | In Spring apps      | Framework-agnostic| Legacy              |

// @Autowired — by TYPE inject karta hai
@Autowired
private PaymentGateway gateway; // Type = PaymentGateway

// @Resource — by NAME inject karta hai

// @Inject — @Autowired jaisa, but required attribute nahi

@Resource legacy code mein milta hai."

Q28. Lazy vs Eager Loading?

// EAGER — Related entities turant load ho jaati hain (join query)
@Entity
public class Order {
    @ManyToOne(fetch = FetchType.EAGER) // Default for @ManyToOne, @OneToOne
    private User user; // Order fetch karte hi User bhi load ho jaata hai
}

// LAZY — Related entities tabhi load hoti hain jab access karo
@Entity
public class User {
    @OneToMany(fetch = FetchType.LAZY) // Default for @OneToMany, @ManyToMany
    private List<Order> orders; // Jab user.getOrders() call karo TABHI load
}

// LAZY LOADING PROBLEM: LazyInitializationException
@Service
public class UserService {
    @Transactional // Transaction yahan khatam ho jaati hai
    public User getUser(Long id) {
        return userRepository.findById(id).orElseThrow();
    }
}

// Controller mein:
User user = userService.getUser(1L);
user.getOrders(); // ❌ LazyInitializationException! Transaction khatam ho gayi

// SOLUTIONS:
// 1. @Transactional controller/service mein extend karo
// 2. @EntityGraph — specific fetch strategy
@EntityGraph(attributePaths = {"orders", "orders.items"}) // Join fetch
@Query("SELECT u FROM User u WHERE u.id = :id")

Q29. N+1 Problem kya hota hai?
------------------------------------------------------------------------

// PROBLEM: 1 query to fetch N users, then N queries to fetch each user's orders
// 1 + N queries = N+1 problem

// BAD CODE:
List<User> users = userRepository.findAll(); // 1 query: SELECT * FROM users
for (User user : users) {
    List<Order> orders = user.getOrders(); // N queries: SELECT * FROM orders WHERE user_id=?

// @Transactional = AOP proxy ke through kaam karta hai
// Method call intercept hoti hai, transaction start/commit/rollback hota hai

@Service
public class OrderService {

    // DEFAULT BEHAVIOR:
    // propagation = REQUIRED (existing transaction join, nahin to naya create)
    // isolation = DEFAULT (DB ka default)
    // rollbackFor = RuntimeException, Error (checked exceptions pe ROLLBACK NAHI)
    // readOnly = false
    @Transactional
    public void placeOrder(Order order) {
        orderRepository.save(order);           // 1
        inventoryService.deduct(order.getItems()); // 2 — exception here
        paymentService.charge(order);           // 3 — rollback hoga, 1 bhi undo
    }

    // CHECKED EXCEPTION pe rollback:
    @Transactional(rollbackFor = Exception.class)
    public void processPayment(Payment payment) throws PaymentException {
        // Ab PaymentException (checked) pe bhi rollback hoga
    }

    // READ-ONLY transaction (faster — no dirty checking)
    @Transactional(readOnly = true)
    public List<User> getAllUsers() {
        return userRepository.findAll();
    }

    // REQUIRES_NEW — new transaction, existing suspend
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void auditLog(String action) {
        // Yeh HAMESHA commit hoga, chahe outer transaction rollback ho
        auditRepository.save(new AuditLog(action));
    }
}

⭐ 6+ Years Interview Answer

"@Transactional works through Spring AOP proxies. 
When a transactional method is invoked through the proxy, Spring begins a transaction before calling the target method.
 If the method completes successfully, the proxy commits the transaction; if a configured exception occurs,
  it rolls the transaction back. By default, only unchecked exceptions (RuntimeException and Error) trigger rollback.
   Internal self-invocation bypasses the proxy,
 so transactional behavior is not applied unless the call goes through another Spring-managed proxy."

Q32. JWT kya hota hai aur kaise use hota hai?
------------------------------------------------------------------------

JWT (JSON Web Token) = Stateless authentication token
Structure: header.payload.signature (dot separated, Base64 encoded)

HEADER:  {"alg": "HS256", "typ": "JWT"}
PAYLOAD: {"sub": "user123", "roles": ["ADMIN"], "exp": 1234567890}
SIGNATURE: HMACSHA256(base64(header) + "." + base64(payload), secret)

FLOW:
1. User login kare (POST /auth/login)
2. Server credentials verify kare
3. JWT token generate karke return kare
4. Client har request mein token bheje (Authorization: Bearer <token>)
5. Server token verify kare — NO DB CALL (stateless!)
6. Expiry check kare

// SPRING BOOT JWT IMPLEMENTATION:
// pom.xml: io.jsonwebtoken:jjwt-api, jjwt-impl, jjwt-jackson

@Component
public class JwtUtil {
    @Value("${jwt.secret}")
    private String secret; // application.properties se

    @Value("${jwt.expiration:86400000}") // 24 hours default
    private long expiration;

    // Token generate karo
    public String generateToken(UserDetails userDetails) {
        return Jwts.builder()
            .setSubject(userDetails.getUsername())
            .claim("roles", userDetails.getAuthorities()) // Custom claims
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + expiration))
            .signWith(getSigningKey(), SignatureAlgorithm.HS256)
            .compact();
    }

    // Token se username nikalo
    public String extractUsername(String token) {
        return extractClaim(token, Claims::getSubject);
    }

    // Token valid hai?
    public boolean validateToken(String token, UserDetails userDetails) {
        String username = extractUsername(token);
        return username.equals(userDetails.getUsername()) && !isTokenExpired(token);
    }

    private boolean isTokenExpired(String token) {
        return extractClaim(token, Claims::getExpiration).before(new Date());
    }

    private <T> T extractClaim(String token, Function<Claims, T> claimsResolver) {
        Claims claims = Jwts.parserBuilder()
            .setSigningKey(getSigningKey())
            .build()
            .parseClaimsJws(token)
            .getBody();
        return claimsResolver.apply(claims);
    }

    private Key getSigningKey() {
        return Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    }
}

// JWT Filter — har request pe token check karo
@Component
public class JwtAuthFilter extends OncePerRequestFilter {
    @Autowired private JwtUtil jwtUtil;
    @Autowired private UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        // Header se token nikalo
        String header = request.getHeader("Authorization");
        String token = null;
        String username = null;

        if (header != null && header.startsWith("Bearer ")) {
            token = header.substring(7); // "Bearer " remove karo
            try {
                username = jwtUtil.extractUsername(token);
            } catch (JwtException e) {
                response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Invalid token");
                return;
            }
        }

        // SecurityContext mein set karo (agar not already authenticated)
        if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
            UserDetails userDetails = userDetailsService.loadUserByUsername(username);
            if (jwtUtil.validateToken(token, userDetails)) {
                UsernamePasswordAuthenticationToken auth =
                    new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities());
                auth.setDetails(new WebAuthenticationDetailsSource().buildDetails(request));
                SecurityContextHolder.getContext().setAuthentication(auth);
            }
        }
        chain.doFilter(request, response);
    }
}

------------------------------------------------------------------------

CONCEPT: Cross-cutting concerns (logging, security, transactions) ko business logic se alag karo

// WITHOUT AOP — REPEATED CODE EVERYWHERE:

// WITH AOP — Clean business logic:
@Service
public class UserService {
    public void createUser(User user) {
        userRepository.save(user); // ONLY business logic!
    }
}

// Aspect — cross-cutting concern yahan:
@Aspect
@Component
public class LoggingAspect {
    private static final Logger log = LoggerFactory.getLogger(LoggingAspect.class);

    // POINTCUT — kaunse methods pe apply karna hai
    @Around("execution(* com.myapp.service.*.*(..))") // Saari service methods
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        String method = joinPoint.getSignature().getName();
        Object[] args = joinPoint.getArgs();

        log.info("Method {} called with args: {}", method, Arrays.toString(args));
        long start = System.currentTimeMillis();

        try {
            Object result = joinPoint.proceed(); // Actual method call karo
            long time = System.currentTimeMillis() - start;
            log.info("Method {} completed in {}ms", method, time);
            return result;
        } catch (Exception ex) {
            log.error("Method {} failed: {}", method, ex.getMessage());
            throw ex;
        }
    }

    // @Before — method se pehle
    @Before("@annotation(com.myapp.annotation.RequiresAuth)")
    public void checkAuth(JoinPoint jp) {
        // Authentication check
    }

    // @AfterReturning — successful return ke baad
    @AfterReturning(pointcut = "execution(* saveUser(..))", returning = "result")
    public void afterSave(Object result) {
        // Audit log
    }

    // @AfterThrowing — exception ke baad
    @AfterThrowing(pointcut = "execution(* processPayment(..))", throwing = "ex")
    public void onPaymentFailure(Exception ex) {
        // Alert system
    }
}

// Custom annotation example:
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Timed { }

@Around("@annotation(Timed)")
public Object time(ProceedingJoinPoint pjp) throws Throwable {
    long start = System.currentTimeMillis();
    Object result = pjp.proceed();
    System.out.println("Time: " + (System.currentTimeMillis() - start) + "ms");
    return result;
}

Q37. Circular Dependency kaise resolve hoti hai?

// CIRCULAR DEPENDENCY: A → B → A (A needs B, B needs A)

@Service
public class ServiceA {
    @Autowired private ServiceB b; // A needs B
}
@Service
public class ServiceB {
    @Autowired private ServiceA a; // B needs A
}

// Field/setter injection mein: Spring 2.6 se pehle — silently handled (3-step resolution)

// SOLUTIONS:
// 1. Refactor — circular dependency = design problem! Extract common logic to ServiceC

// 2. @Lazy on one side
@Service
public class ServiceA {
    @Autowired @Lazy // ServiceB late initialize hoga — circular resolve
    private ServiceB b;
}

// 3. Setter injection (one side)
@Service
public class ServiceA {
    private ServiceB b;
    @Autowired
    public void setServiceB(ServiceB b) { this.b = b; }
}

// 4. ApplicationContext se getBean() (runtime)

public List<Order> getOrders(Long userId) {
    return orderClient.getOrdersByUser(userId);
}

public List<Order> defaultOrders(Long userId, Exception ex) {
    log.warn("Order service unavailable, returning empty list");
    return Collections.emptyList();
}

Spring Microservices Communication - Detailed Interview Notes
1. OpenFeign (Synchronous Communication)
Concept:
OpenFeign is a declarative HTTP client. You only define an interface, and Spring generates a proxy implementation. 
When your code calls the interface method, it is actually calling a Spring-generated proxy. 
The proxy converts the Java method into an HTTP request, sends it to the target microservice, waits for the response,
 converts JSON back into Java objects using Jackson, and returns the result.


Internal Flow:
Client
 ↓
Controller
 ↓
Service
 ↓
Feign Proxy (generated by Spring)
 ↓
HTTP Request
 ↓
Order Service
 ↓
HTTP Response (JSON)
 ↓
Jackson converts JSON -> Java Objects
 ↓
Feign Proxy returns object
 ↓
Service -> Controller -> Client
Why use it?
- No RestTemplate boilerplate
- Interface-based coding
- Integrates with Eureka, LoadBalancer and Circuit Breaker.
Important Properties:
•	order.service.url=http://localhost:8082
•	spring.cloud.openfeign.client.config.default.connect-timeout=3000
•	spring.cloud.openfeign.client.config.default.read-timeout=5000
•	spring.cloud.openfeign.client.config.default.logger-level=basic
Interview Answer:
Feign is synchronous. The caller blocks until the remote service responds. Internally Spring creates a proxy for the interface and the proxy performs HTTP communication.
2. Kafka (Asynchronous Communication)
Concept:
Kafka follows publish-subscribe messaging. The producer publishes an event to a topic and immediately continues processing. It does not wait for consumers. Consumers independently read events from the topic using their consumer group offsets.
Internal Flow:
Order Service
 ↓
KafkaTemplate.send()
 ↓
Producer Serializes Event -> JSON
 ↓
Kafka Broker
 ↓
Topic: order-created
 ↓
Consumer Group
 ↓
Notification Service
Inventory Service
Analytics Service
 ↓
@KafkaListener invoked automatically
Why use Kafka?
Loose coupling, retries, multiple consumers, high throughput, event-driven architecture.
Important Properties:
•	spring.kafka.bootstrap-servers=localhost:9092
•	spring.kafka.consumer.group-id=notification-service
•	spring.kafka.consumer.auto-offset-reset=earliest
•	spring.kafka.producer.key-serializer=StringSerializer
•	spring.kafka.producer.value-serializer=JsonSerializer
•	spring.kafka.consumer.value-deserializer=JsonDeserializer
Interview Point:
Producer never knows who consumes the message. Multiple services can subscribe to the same topic independently.
3. Circuit Breaker (Resilience4j)
Concept:
A Circuit Breaker protects your application from repeatedly calling an unhealthy downstream service. It works through Spring AOP. Instead of calling the real service directly, requests first pass through a proxy that tracks failures.
Internal Flow:
Client
 ↓
Circuit Breaker Proxy
 ↓
Remote Service

If failures exceed threshold:
Circuit -> OPEN
 ↓
Remote call skipped
 ↓
Fallback Method

After wait duration:
HALF_OPEN
 ↓
Few trial requests
 ↓
Success -> CLOSED
Failure -> OPEN
Configuration:
•	resilience4j.circuitbreaker.instances.orderService.failure-rate-threshold=50
•	resilience4j.circuitbreaker.instances.orderService.sliding-window-size=10
•	resilience4j.circuitbreaker.instances.orderService.wait-duration-in-open-state=10s
•	resilience4j.circuitbreaker.instances.orderService.permitted-number-of-calls-in-half-open-state=3
Interview Explanation:
Closed = normal traffic. Open = fail fast and execute fallback. Half-Open = limited requests to test recovery.
Comparison
Feature	Feign	Kafka	Circuit Breaker
Communication	Sync	Async	Protection Pattern
Waits?	Yes	No	Depends
Transport	HTTP	Message Broker	AOP Proxy
Use Case	Immediate Response	Events	Fault Tolerance

public class PaymentService {

        // STRUCTURED LOGGING — JSON format better for ELK/Splunk

            gateway.charge(payment);

// management.endpoint.health.show-details=always

// CUSTOM METRICS:
@Service
public class OrderService {
    private final Counter orderCounter;
    private final Timer orderProcessingTimer;

    public OrderService(MeterRegistry registry) {
        this.orderCounter = registry.counter("orders.created.total");
        this.orderProcessingTimer = registry.timer("orders.processing.time");
    }

    public Order placeOrder(Order order) {
        return orderProcessingTimer.recordCallable(() -> {
            Order result = processOrder(order);
            orderCounter.increment();
            return result;
        });
    }
}

EXAMPLE — @Scheduled:
@Component
@EnableScheduling
public class CleanupJob {

    // Har din raat 2 baje run karo
    @Scheduled(cron = "0 0 2 * * *")
    public void cleanOldData() {
        dataRepository.deleteOlderThan(LocalDate.now().minusDays(90));
    }

    // Har 30 second mein
    @Scheduled(fixedDelay = 30000) // Previous execution end se 30 sec baad
    public void syncCache() { cacheService.refresh(); }
}

EXAMPLE — @TransactionalEventListener:
@Service
public class OrderService {
    @Autowired ApplicationEventPublisher publisher;

    @Transactional
    public void placeOrder(Order order) {
        orderRepository.save(order);
        // Transaction commit ke BAAD event fire hoga (AFTER_COMMIT)
        // Agar exception aayi → event fire nahi hoga → consistent!
        publisher.publishEvent(new OrderCreatedEvent(order));
    }
}

@Component
public class OrderEventHandler {
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void handleOrderCreated(OrderCreatedEvent event) {
        // Kafka mein publish karo — transaction committed hai confirm
        kafkaTemplate.send("orders", event.getOrder());
    }
}

@Primary vs @Qualifier      → Primary = global default, Qualifier = specific override
