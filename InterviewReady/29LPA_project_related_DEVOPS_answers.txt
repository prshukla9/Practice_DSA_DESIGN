SECTION 1: DOCKER — CONTAINERS KYON? KYA? KAISE?
================================================================================

Q1. Docker kya hai? Virtual Machine se kaise alag?

STORY: Delivery analogy
PEHLE (Without Docker):
  Dev: "Meri machine pe kaam karta hai!"
  Ops: "Production pe crash ho raha hai!"
  REASON: Different Java version, different OS, different env vars

DOCKER SOLUTION:
  "Application + uski dependencies + configuration = EK package (container)"
  Jaise courier box — andar sab kuch packed, kahin bhi deliver karo

DOCKER vs VIRTUAL MACHINE:
  VM:        [App] [OS] | [App] [OS]  →  Hypervisor  →  Host OS
             Heavy, slow start (mins), GBs of size
  Container: [App] [App] [App]        →  Docker Engine  →  Host OS
             Light, fast start (seconds), MBs of size

KEY DOCKER CONCEPTS:
  Image     = Blueprint (like Java .class file — bana ke rakh do)
  Container = Running instance (like JVM pe running Java process)
  Dockerfile = Instructions to build image
  Registry  = Docker Hub / ECR — images store karo
  Volume    = Persistent storage (container restart pe data survive)
  Network   = Containers ke beech communication

// ============================================================
// SINGLE-STAGE DOCKERFILE (Basic — interviews mein pucha jaata)
// ============================================================

# Dockerfile (single stage — simple but inefficient)
FROM eclipse-temurin:21-jdk-alpine

# Working directory container ke andar
WORKDIR /app

# JAR copy karo
COPY target/healthcare-service-1.0.0.jar app.jar

# Port expose karo (documentation only — actual mapping docker run pe)
EXPOSE 8080

# Container start hone pe yeh command chalao
ENTRYPOINT ["java", "-jar", "app.jar"]

// ============================================================
// MULTI-STAGE DOCKERFILE (Production — yeh discuss karo interview mein)
// ============================================================

# Stage 1: BUILD — Maven compile + package
# Full JDK needed — source code compile karna hai
FROM maven:3.9.4-eclipse-temurin-21 AS builder

WORKDIR /build

# TRICK: pehle dependencies download karo (layer cache!)
# Dependencies kam change hoti hain — yeh layer cache mein rehta hai
COPY pom.xml .
RUN mvn dependency:go-offline -q

# Ab source code copy karo (frequently changes)
COPY src ./src

# Build — skip tests (CI mein already run ho chuke)
RUN mvn clean package -DskipTests

# --------------------------------------------------------
# Stage 2: RUNTIME — Sirf JRE, no Maven, no source code
# Final image mein JDK nahi chahiye → smaller + secure
FROM eclipse-temurin:21-jre-alpine AS runtime

# Security best practice: root mat chalaao
RUN addgroup -S healthcare && adduser -S appuser -G healthcare
USER appuser

WORKDIR /app

# ONLY the JAR from builder stage — baki sab throw away
COPY --from=builder /build/target/healthcare-service-*.jar app.jar

# Health check — Docker ko pata chale app alive hai
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:8080/actuator/health || exit 1

EXPOSE 8080

# JVM tuning for containers:
# -XX:+UseContainerSupport  = Docker memory limits respect karo (Java 10+)
# -XX:MaxRAMPercentage=75   = Container RAM ka 75% use karo
# -XX:+ExitOnOutOfMemoryError = OOM pe crash (Kubernetes restart karega)
ENTRYPOINT ["java", \
  "-XX:+UseContainerSupport", \
  "-XX:MaxRAMPercentage=75.0", \
  "-XX:+ExitOnOutOfMemoryError", \
  "-Djava.security.egd=file:/dev/./urandom", \
  "-jar", "app.jar"]

BENEFIT OF MULTI-STAGE:
  Single stage:  ~500MB image (JDK + Maven + source + JAR)
  Multi-stage:   ~80MB image  (JRE + JAR only)
  Less attack surface → More secure → Faster pull in Kubernetes

SCENARIO QUESTION: "Kya docker image size matter karta hai?"
"Haan! 80MB vs 500MB:
 - CI/CD: Har deployment pe pull time → faster
 - K8s: Node pe image cache → startup time kam
 - Security: Choti image = kam vulnerabilities"

SECTION 4: KUBERNETES — PRODUCTION ORCHESTRATION
================================================================================

Q2. Kubernetes kya karta hai? Aur kyon Docker Compose nahi?

STORY: Restaurant analogy
  Docker = Chef (ek dish banana jaanta hai)
  Kubernetes = Restaurant Manager:
    - "Table 5 ke liye 3 chefs chahiye" → Auto-scaling
    - "Chef 2 beemar hai" → Self-healing (restart)
    - "New menu item add karo, purana dhire dhire hata do" → Rolling updates

KUBERNETES COMPONENTS:
  Pod        = Smallest unit — ek ya adhik containers
  Deployment = "Hamesha N pods chalao, aur aise update karo" — replicas, strategy
  Service    = Stable DNS name + Load balancer for pods
  ConfigMap  = Non-secret config (DB URL, feature flags)
  Secret     = Sensitive data (passwords, API keys) — base64 encoded
  Ingress    = HTTP routing (path-based, host-based) — single entry point
  HPA        = Horizontal Pod Autoscaler — CPU/memory pe scale

        app: patient-service

    spec:
      containers:
      - name: patient-service
        image: 123456789.dkr.ecr.us-east-1.amazonaws.com/patient-service:v1.2.3
        imagePullPolicy: Always  # Hamesha registry se pull karo

        ports:
        - containerPort: 8080

        # ENV from ConfigMap
        envFrom:
        - configMapRef:
            name: patient-service-config

        # ENV from Secret
        env:
        - name: SPRING_DATASOURCE_PASSWORD
          valueFrom:
            secretKeyRef:
              name: patient-service-secrets
              key: DB_PASSWORD
        - name: OPENAI_API_KEY
          valueFrom:
            secretKeyRef:
              name: patient-service-secrets
              key: OPENAI_API_KEY

        # Resource limits — CRITICAL for K8s scheduling
        resources:
          requests:              # "Mujhe itna chahiye" — scheduling ke liye
            memory: "512Mi"
            cpu: "250m"          # 250 millicores = 0.25 CPU
          limits:                # "Itne se zyada nahi lunga" — OOM prevention

            cpu: "1000m"         # 1 full CPU

        # Liveness probe — "Is app alive? Agar nahi → restart"
        livenessProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 60   # App startup time do
          periodSeconds: 30
          failureThreshold: 3

        # Readiness probe — "Is app ready for traffic?"
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
          failureThreshold: 3

// ============================================================
// Jenkinsfile — Declarative Pipeline
// ============================================================

pipeline {
    // Kahan run karo — Docker container mein (clean environment)
    agent {
        docker {
            image 'maven:3.9.4-eclipse-temurin-21'
            args '-v /root/.m2:/root/.m2'   // Maven cache mount
        }
    }

    // Trigger: Git push pe auto-run
    triggers {
        githubPush()    // GitHub webhook se trigger
    }

    environment {
        // Jenkins credentials store se securely fetch karo
        DOCKER_REGISTRY = "123456789.dkr.ecr.us-east-1.amazonaws.com"
        IMAGE_NAME      = "patient-service"
        IMAGE_TAG       = "${GIT_COMMIT[0..7]}"   // Short git hash = unique tag
        SONAR_TOKEN     = credentials('sonar-token')
        AWS_CREDS       = credentials('aws-ecr-credentials')
    }

    stages {

        stage('Checkout') {
            steps {
                checkout scm    // Git se latest code fetch
                echo "Building commit: ${env.GIT_COMMIT}"
            }
        }

        stage('Unit Tests') {
            steps {
                sh 'mvn test -pl patient-service'
            }
            post {
                always {
                    // Test results publish — Jenkins UI mein dikhe
                    junit 'patient-service/target/surefire-reports/*.xml'
                    // JaCoCo coverage report
                    jacoco(
                        execPattern: '**/target/*.exec',
                        minimumLineCoverage: '80'   // 80% se kam → FAIL
                    )
                }
            }
        }

        stage('Code Quality') {
            steps {
                // SonarQube analysis — code smells, vulnerabilities
                sh """
                    mvn sonar:sonar \
                    -Dsonar.projectKey=healthcare-patient-service \
                    -Dsonar.host.url=http://sonarqube:9000 \
                    -Dsonar.login=${SONAR_TOKEN}
                """
            }
        }

        stage('Build & Package') {
            steps {
                sh 'mvn clean package -DskipTests -pl patient-service'
            }
        }

        stage('Docker Build') {
            steps {
                script {
                    // Docker image build karo
                    sh "docker build -t ${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG} ./patient-service"
                    sh "docker tag ${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG} ${DOCKER_REGISTRY}/${IMAGE_NAME}:latest"
                }
            }
        }

        stage('Security Scan') {
            steps {
                // Trivy — Docker image vulnerability scan
                sh """
                    trivy image \
                    --exit-code 1 \
                    --severity HIGH,CRITICAL \
                    ${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}
                """
                // HIGH/CRITICAL vulnerability pe build fail karo
                // Healthcare compliance ke liye CRITICAL!
            }
        }

        stage('Push to ECR') {
            steps {
                script {
                    // AWS ECR login (Amazon's Docker registry)
                    sh "aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin ${DOCKER_REGISTRY}"
                    sh "docker push ${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}"
                    sh "docker push ${DOCKER_REGISTRY}/${IMAGE_NAME}:latest"
                }
            }
        }

        stage('Deploy to Staging') {
            steps {
                // kubectl image update karo — rolling deployment
                sh """
                    kubectl set image deployment/patient-service \
                    patient-service=${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG} \
                    -n healthcare-staging
                """
                // Deploy complete hone ka wait karo
                sh "kubectl rollout status deployment/patient-service -n healthcare-staging --timeout=5m"
            }
        }

        stage('Integration Tests') {
            steps {
                // Staging pe run karo
                sh "mvn verify -Pintegration-tests -Dbase.url=https://staging-api.healthcare.com"
            }
        }

        stage('Deploy to Production') {
            // MANUAL APPROVAL for Production!
            input {
                message "Deploy to PRODUCTION?"
                ok "Yes, deploy!"
                submitter "team-lead,devops-admin"   // Only these can approve
            }
            steps {
                sh """
                    kubectl set image deployment/patient-service \
                    patient-service=${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG} \
                    -n healthcare-prod
                """
                sh "kubectl rollout status deployment/patient-service -n healthcare-prod --timeout=10m"
            }
        }
    }

    post {
        success {
            // Slack notification success
            slackSend(
                channel: '#deployments',
                color: 'good',
                message: "✅ ${IMAGE_NAME}:${IMAGE_TAG} deployed to PROD successfully"
            )
        }
        failure {
            slackSend(
                channel: '#deployments',
                color: 'danger',
                message: "❌ Pipeline FAILED for ${IMAGE_NAME} — ${BUILD_URL}"
            )
            // Automatic rollback!
            sh "kubectl rollout undo deployment/patient-service -n healthcare-prod || true"
        }
        always {
            // Cleanup
            sh "docker rmi ${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG} || true"
        }
    }
}

SCENARIO QUESTION: "Deploy ke baad production mein issue aaya — kya karoge?"
"Jenkins pipeline failure → automatic rollback (`kubectl rollout undo`)
 Phir: kubectl rollout history deployment/patient-service (version history dekho)
 Phir manually specific version pe rollback: kubectl rollout undo --to-revision=3"

WHY EKS vs. Self-managed K8s:
- Control plane AWS manage karta hai (upgrades, HA, etcd backups)
- AWS-native: IAM for K8s RBAC, ELB auto-integration, EBS volumes
- IRSA (IAM Roles for Service Accounts) — pod-level AWS permissions

- Managed: Automated backups (point-in-time recovery — 35 days)
- Multi-AZ: Automatic failover (< 60 sec) — Healthcare needs this!
- Read Replicas: Vector search heavy → separate replica for AI queries

    /**
     * Lab report PDF upload karo
     * Encrypted at rest (SSE-S3 automatic), encrypted in transit (HTTPS)
     */
    public String uploadMedicalDocument(String patientId, byte[] pdfBytes) {
        String key = String.format("patients/%s/reports/%s.pdf",
            patientId, UUID.randomUUID());

        // Server-side encryption mandatory for HIPAA
        PutObjectRequest req = PutObjectRequest.builder()
            .bucket(bucketName)
            .key(key)
            .serverSideEncryption(ServerSideEncryption.AES256)  // SSE-S3
            .contentType("application/pdf")
            .build();

        s3.putObject(req, RequestBody.fromBytes(pdfBytes));
        return key;
    }

    /**
     * Time-limited presigned URL — Direct patient download
     * URL 15 min mein expire ho jaata hai — security!
     */
    public String generatePresignedUrl(String documentKey) {
        S3Presigner presigner = S3Presigner.create();

        GetObjectPresignRequest presignReq = GetObjectPresignRequest.builder()
            .signatureDuration(Duration.ofMinutes(15))   // 15 min only!
            .getObjectRequest(r -> r.bucket(bucketName).key(documentKey))
            .build();

        return presigner.presignGetObject(presignReq).url().toString();
    }
}

SCENARIO 1: "Production database slow hai — first response kya hoga?"
"Steps:
 1. CloudWatch RDS metrics check: CPU, connections, read/write latency
 2. Slow query log enable: pg_stat_statements se slow queries identify
 3. Kill long-running queries jo block kar rahe hain
 4. Quick wins: Missing index? EXPLAIN ANALYZE run karo
 5. If load: Read replica ko vector search ke liye use karo
 6. If still issue: Vertical scale RDS (maintenance window ke baad)"

SCENARIO 2: "Docker image ECR mein push karo but size 2GB hai"
"2GB bahut zyada hai! Debug karo:
 docker history patient-service:latest -- Har layer ka size dekho
 Usually: Maven dependencies layer (>1GB) skip nahi kar rahe
 FIX: Multi-stage build use karo!
   Builder stage: JDK + Maven + source → only JAR copy karo runtime mein
   Runtime stage: Sirf JRE + JAR → ~80MB
 Also: .dockerignore file add karo:
   target/, .git/, *.md — unnecessary files mat copy karo"

SCENARIO 3: "Kubernetes pod OOMKilled ho raha hai"
"OOMKilled = container ne memory limit exceed ki, K8s ne kill kar diya
 Steps:
 1. kubectl describe pod <name> -- 'OOMKilled' reason confirm karo
 2. Memory usage check: kubectl top pod <name>
 3. JVM tuning check: -XX:MaxRAMPercentage=75 laga hai?
 4. Heap dump leke analyze karo: -XX:+HeapDumpOnOutOfMemoryError
 5. Short term: limits.memory badhao (512Mi → 1Gi)
 6. Long term: Memory leak fix karo (heap dump analysis)"

SCENARIO 4: "Jenkins deployment fail hua production mein — next steps?"
"Automatic rollback Jenkins post { failure } block ne kar diya (kubectl rollout undo)
 Investigate:
 1. Jenkins log dekho — kaunsa stage fail hua?
 2. kubectl logs — application error kya tha?
 3. kubectl describe pod -- CrashLoopBackOff? OOMKilled? ImagePullError?
 4. Common causes:
    - Startup fail: Missing env var? DB connection? Kafka not reachable?
    - ImagePullError: ECR credentials expire? Wrong tag?
    - Crash on start: Application exception? Config error?
 5. Fix karo, PR raise karo, pipeline re-run karo"

SCENARIO 5: "Healthcare app ka compliance audit — AWS mein kya show karte?"
"HIPAA compliance evidence:
 1. Encryption at rest: RDS encryption ON, S3 SSE-S3/KMS, EBS encrypted
 2. Encryption in transit: All internal traffic TLS (MSK SASL_SSL, Redis SSL)
 3. Access control: IAM least privilege, K8s RBAC, VPC isolation
 4. Audit logging: CloudTrail (API calls), CloudWatch Logs (application)
 5. Backup: RDS automated backups 35 days, Multi-AZ, point-in-time recovery
 6. Monitoring: CloudWatch alarms, GuardDuty (threat detection)
 7. Penetration testing: Trivy image scan in CI/CD pipeline"
